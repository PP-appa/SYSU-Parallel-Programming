## 2. 实验过程和核心代码

### 2.1 实验环境准备
在 Ubuntu 环境下配置开发工具链，确认 gcc/g++ 安装无误（版本 11.4.0）。同时，针对最后的对比实验部分，通过包管理器安装了 `intel-mkl` 库。所有矩阵操作均统一使用尺寸 $N = 1024$ 的方阵进行测试与计时。

### 2.2 版本 1：Python 基础算法
作为对比实验的基准线下限阶段，首先实现了原生的 Python 矩阵乘法。由于原生 Python List 是基于对象的数组引用，内存并不连续，同时每一次内层循环的迭代都面临解释器的严重开销。

**核心代码:**
```python
for i in range(m):
    for j in range(n):
        for inner_k in range(k):
            C[i][j] += A[i][inner_k] * B[inner_k][j]
```

### 2.3 版本 2：C/C++ 朴素实现 
在 C/C++ 环境下，为了避开二维指针对性能和内存的随机影响，实验中采用一维数组来模拟二维矩阵操作，通过 `[i * n + j]` 维持内存空间的连续分布。本阶段代码采用了最基础的 $O(N^3)$ 原生三层循环。

**核心代码:**
```c
for (int i = 0; i < m; i++) {
    for (int j = 0; j < n; j++) {
        for (int inner_k = 0; inner_k < k; inner_k++) { 
            C[i * n + j] += A[i * k + inner_k] * B[inner_k * n + j];
        }
    }
}
```

### 2.4 版本 3：C/C++ 循环展开优化
实验通过探索循环顺序重排证明了缓存（Cache）命中率的重要性。C 语言的数据服从行优先（Row-Major）存储原则，将 `j` 迭代置于最内层可以保证程序在访问数组 `B` 和数组 `C` 时具备空间局部性。

**核心代码:**
```c
for (int i = 0; i < m; i++) {
    for (int inner_k = 0; inner_k < k; inner_k++) {
        for (int j = 0; j < n; j++) { 
            C[i * n + j] += A[i * k + inner_k] * B[inner_k * n + j];
        }
    }
}
```

### 2.5 版本 4 与 5：编译期优化及循环展开
**编译参数优化 (-O3)：** 直接使能 gcc 工具链自带的深层优化策略，编译器会自动进行指令重排和初步的向量化改写。
**循环展开探索 (Unroll)：** 本阶段手工将最内层循环展开以减少跳转指令和分支判断代价。在这个环节中，我尝试对 `-O0` 等级下的循环展开进行了考察，测试表明仅仅展开最内层循环而不用高级编译参数，在同一份代码下能将耗时从 5.18 秒缩减到 1.76 秒；这实证了除了编译器的向量化操作外，消减循环跳转带来的流水线破坏、进而暴露出更多指令级并行度（ILP），也是影响计算密集型作业的瓶颈之一。

**核心代码 (循环展开结合行顺序):**
```c
double r = A[i * k + inner_k]; 
int j = 0;
for (; j <= n - 4; j += 4) { 
    C[i * n + j]     += r * B[inner_k * n + j];
    C[i * n + j + 1] += r * B[inner_k * n + j + 1];
    C[i * n + j + 2] += r * B[inner_k * n + j + 2];
    C[i * n + j + 3] += r * B[inner_k * n + j + 3];
}
// 后续剩余非对齐列处理循环（省略）
```

### 2.6 版本 6：调用 Intel MKL
实验利用了 Intel 提供的数学核心函数库，通过引入头文件链接 `-lmkl_rt` 搭配专属的内存分配 `mkl_malloc` 来进行矩阵计算。它对底层指令和多线程并发调度进行了完全的重构封装，是工业标准。

**核心代码:**
```c
cblas_dgemm(CblasRowMajor, CblasNoTrans, CblasNoTrans,
            m, n, k, 1.0, A, k, B, n, 0.0, C, n);
```

## 3. 实验结果
测试条件：三组输入维度 $m, n, k$ 均为 1024。
双精度浮点计算量理论推导：约 $2N^3 = 2.147 \times 10^9$ 次浮点计算操作。

| 版本 | 实现描述 | 运行时间(sec.) | 相对加速比 | 绝对加速比 | 浮点性能(GFLOPS) | 峰值性能百分比 |
|:---:|:---|:---:|:---:|:---:|:---:|:---:|
| 1 | Python | 86.500 | - | 1.00x | 0.024 | 0.03% |
| 2 | C/C++ 基础 | 6.123 | 14.12x | 14.12x | 0.350 | 0.42% |
| 3 | 调整循环顺序 | 2.129 | 2.87x | 40.62x | 1.008 | 1.21% |
| 4 | 编译优化(-O3结合版2) | 3.005 | 0.70x | 28.78x | 0.714 | 0.86% |
| 5 | 循环展开(+O3+顺序) | 0.195 | 15.41x | 443.58x | 11.012 | 13.24% |
| 6 | Intel MKL | 0.031 | 6.29x | 2790.3x | 69.273 | 83.26% |

*(表格注：相对加速比为当前版本对比上一版本所取得的速度增益；由于实现 4 基于未更改顺序版 2 构建，而实现 3 是顺序修改版，此时时间不升反弱属于预期现象方案的分支)*

## 4. 实验感想

整个实验通过由浅入深的步骤逐步搭建并测试同一程序的优化迭代，直观地表现出高级语言和底层实现方式、结构策略之间的运行开销存在巨大的体量差距。

其中令我收获颇深的是探讨在关闭 `-O3` 优化的前提下，依然使用循环手写展开所带来的巨幅提速。起初我认为必须配合编译器的向量化，循环展开的意义才显著，但在和教师进行思路探讨并加以数据验证后我发现：不依赖于向量机件，在减少常规程序中的程序流控制（如条件判定、跳转）从而释放了处理器核心层的流水线资源和标量级 ILP后，代码的效率依然有着明显的改观，这对于掌握底层硬件构造极其关键。

此外在 Ubuntu 物理环境下初次调用外部大内存运算库（Intel MKL）暴露了一些工程方面链接和配置的错误，如出现库指令 `make` 阶段找不到引用的问题；而后学会通过设置专门参数 `-I` 和 `-lmkl_rt` 构建路径的方法。本次验证为往后的并行工具如 MPI 环境铺垫了很好的思路基础。

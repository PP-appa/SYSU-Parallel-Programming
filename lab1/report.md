## 2. 实验过程与核心代码

### 2.1 MPI 主从模式 (Master-Worker) 设计
本实验采用了经典的 Master-Worker 架构：
- **Master 进程 (Rank 0)**：负责随机初始化整个矩阵 $A$ 和 $B$。记录起始时间后，计算出每个 Worker 应当分担的 $A$ 的行数任务，将整份矩阵 $B$ 广播（通过发送点对点消息）给所有 Worker，再将矩阵 $A$ 按行切块发送。待 Worker 计算完毕后，Master 负责接收所有的结果块并无缝拼装为完整的最终矩阵 $C$。
- **Worker 进程 (Rank 1 ~ Size-1)**：在收到 Master 传来的矩阵 $B$ 以及自己负责的那一部分 $A$ 子块后，进行局部的矩阵乘法计算（为了保证性能，沿用了 Lab 0 中的 `i-k-j` 循环顺序优化 Cache 局部性），最后将局部计算结果 $C_{local}$ 发送回 Master。

### 2.2 核心代码解析：Master 任务分发与收集

在分配任务时，为实现较好的均衡性，首先基础按 `m / (size - 1)` 划分行数。对于无法整除的剩余行数，将其全部分配给最后一位 Worker：
```cpp
// 任务分发阶段 (Master)
int rows_per_worker = m / (size - 1); 
for (int src = 1; src < size; ++src) {
    int start_row = (src - 1) * rows_per_worker;
    int num_rows = (src == size - 1) ? (m - start_row) : rows_per_worker;
    // 将整个矩阵B发送给Worker
    MPI_Send(B.data(), n * k, MPI_DOUBLE, src, 0, MPI_COMM_WORLD);
    // 将矩阵A对应的切片发送给Worker
    MPI_Send(A.data() + start_row * n, num_rows * n, MPI_DOUBLE, src, 0, MPI_COMM_WORLD);
}

// ... 

// 结果收集阶段 (Master)
for (int src = 1; src < size; ++src) {
    int start_row = (src - 1) * rows_per_worker;
    int num_rows = (src == size - 1) ? (m - start_row) : rows_per_worker;
    // 直接复用 C 的连续一维地址空间进行接收，避免二次内存拷贝
    MPI_Recv(C.data() + start_row * k, num_rows * k, MPI_DOUBLE, src, 0, MPI_COMM_WORLD, MPI_STATUS_IGNORE);
}
```

## 4. 优化方向讨论

### 4.1 在内存有限的情况下，如何进行大规模矩阵乘法计算？
**答：** 
当矩阵极大时，无论是主进程还是子进程的寻址空间都无法一次性完整存下参与运算的 $A$ 和 $B$，这时候必须引入**分块矩阵乘法（Block Matrix Multiplication）**。
1. **分块读取与计算（Out-of-core Computing）：** 将大矩阵按块（Block）存储在外存硬盘上，或者分布式地存在其余的服务器中。Master 不再一次性加载全量规模，而是每次只从硬盘上读取适合 Worker 物理内存大小的小矩阵块（例如子矩阵 $A_{ij}$ 和 $B_{jk}$）进行发送与对应的运算累加。
2. **掩盖 IO 与通信开销：** 由于分块分发次数翻倍，用阻塞式的 `MPI_Send` 频繁传送小块数据或等待硬盘读取会极大带来流水线停顿。我们可以使用**双缓冲（Double Buffering）或非阻塞异步通信（如 `MPI_Isend` 和 `MPI_Irecv`）**，在 Worker 正在利用本地缓存计算当前块时，由硬件在后台异步收发下一个周期需要运算的数据，使计算（Compute）和通信（Communication）时间互相掩盖隐藏。

### 4.2 如何提高大规模稀疏矩阵乘法性能？
**答：** 
当矩阵为大规模稀疏矩阵时，计算和存储连续大量的元素 `0` 将会带来严重的时间与空间浪费。针对它，有以下几种常见的并行优化手段（SpGEMM）：
1. **采用压缩存储结构（减少内存与网络开销）：** 不应使用常规稠密二维数组存储。改用如 **CSR（Compressed Sparse Row，压缩稀疏行）**、CSC（压缩稀疏列）或 COO 格式。这种格式仅连续保存矩阵中的非零元素值及其所在位置的指针/索引，可以把内存占用和 MPI 需传输的字节量降到极低。
2. **剔除计算冗余：** 在最内层操作时剔除乘以 `0` 的算术运算；当判定当前非零值的索引错位（即相乘必含因数 0）时立刻跳过，只计算真正的有效值。
3. **负载均衡设计：** 在上述的 Lab 1 实验代码中，我是按照 $A$ 的**“行数范围”**给不同的 Worker 均分任务。但对于极端稀疏矩阵而言，如果依然按行切分，分到密集区域行的 Worker 会承担绝大部分计算量，而分到全 0 空白行的的 Worker 会瞬间进入空闲等待，这会触发计算倾斜和长尾效应。因此，对于稀疏矩阵的并行任务调度，应当**按照“非零元素个数（NNZ, Number of Non-Zeros）”而非行数来均等地切分成不同的 Task 发给子进程**，以确保运行时的负载均衡。
